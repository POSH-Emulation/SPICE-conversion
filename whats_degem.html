What is DEGEM?¶

The Design Environment for Generating Emulation Models is a DARPA-funded research project within the POSH program. The goal of the research is to put together an algorithm stack that produces a behavioral model for an Analog/Mixed Signal design that can be used in an FPGA emulator. The input design format is SPICE netlist and the output is expected to use SystemVerilog’s Real Number Modelling constructs.

Challenges in Automating AMS Model Generators¶

Automated modelling procedures is a well researched topic that yielded both characterization flows and digital implementation technics. The added value of this project is putting together all the block-level modelling methods in the context of a system that contains a diversity of analog and digital sub-circuits. To be able to mix different types of circuits in a single modelling environment, the tool needs to have a combination two features: open and powerful configuration interface and a high level of automated circuit analysis capabilities. Automated circuit recognition is the driving feature of this project. Successful recognition of circuit topology and other information simplifies the manual configuration required by users. The remaining manual input still needs to be supported by automated decision making that should be presented to the user, with level of confidence, to be confirmed or corrected.

Emulation Models are able to copy some aspects of the original designs. Since the goal of the models is to be implemented in configurable hardware, their scope should be limited to the aspects that are consequential to the digital emulation. Other aspects, while important in the context of analog design, should be ignored or otherwise take up precious emulator resources without contributing to the overall coverage of the emulation test. The challenge to DEGEM is to be able to model the right behavioral aspects and avoid the others. At the same time, it should present the user with an easy control of what part of the model are to be included in the final RNM output.

Tool Development Architectural Choices¶

The tool is coded in a command line run script. The command line options are kept to minimal. Each session is saved in a Tcl and JSON hierarchical database to be quickly loaded by another process for continued processing. The tool is constructed to support a cloud-computing usage mode. It can be launched as a TCP server and its disk-saved database allows it to perform short, CGI launched, operations. A graphic user interface is developed in parallel to the tool itself, to be the front-end of the cloud usage mode.
